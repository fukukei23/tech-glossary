TypeScript

JavaScriptに型を追加。フロント/バックエンド両対応

✅ フロントエンド✅ サーバーサイド✅ 型安全⚠ 設定が複雑⚠ ビルドが必要
📌 一言で言うと

「JavaScriptの信頼できる版。Webの実質標準語」

Microsoftが開発。JavaScriptに静的型付けコードを実行する前にデータの型を確認する仕組み。バグを事前に検出できるを加えたスーパーセット。フロントエンド（React/Next.js）からバックエンド（Node.js）、MCPサーバーModel Context Protocol サーバー。ClaudeなどのAIにツールや外部機能を提供する仕組みまで同じ言語で書ける。

🎯 得意な用途

Webフロントエンド：React / Next.js との組み合わせが業界標準。型安全なUIコンポーネントを構築できる。

Node.jsサーバー：Express / Fastify / NestJS などでAPIサーバーやBFFを構築。フロントと言語を統一できるメリットが大きい。

MCPサーバー・CLIツール：Claude Code の拡張（MCPサーバー）の公式SDKがTypeScript製。AIツール開発の主力言語になりつつある。

フルスタック開発：Next.js App Router でフロント・バック・DB操作をすべてTypeScriptで統一する「T3スタック」が人気。

⚖️ 他言語との比較
観点TypeScriptPythonGo

向いている場面Webアプリ・フルスタックAI・データ・PoC高トラフィックAPI
✅ 選ぶ理由 / ⚠ 選ばない理由

選ぶ理由：フロントとバックを同じ言語で書けるためチームの認知コストが下がる。型エラーでバグを事前検出。Webサービス開発では現実的に最有力。

選ばない理由：小さなスクリプトや自動化ツールにはオーバーキル。AIモデル推論など計算集約タスクはPythonの方が圧倒的にライブラリが充実。

💬 よくある会議での質問と答え方
Q.「JavaScriptじゃなくTypeScriptを使う理由は？」
A.「型定義によりIDEの補完が効き、実行前にバグを検出できます。チームが大きくなるほど恩恵が増します。現在のWebプロジェクトでTypeScriptを使わない理由の方が少ないです」
Q.「Next.jsとExpressどちらを選びますか？」
A.「フロント込みのWebサービスならNext.js（App Router）で統一するのが今の主流です。APIだけ提供するマイクロサービスにはExpressやFastifyが軽量で適しています」
